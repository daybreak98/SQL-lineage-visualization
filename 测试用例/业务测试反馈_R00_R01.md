### 业务可用性判断

**结论：适合日常数据开发使用基础阶段，但存在需改进项。**

#### 通过项
1. **元数据仓库设计**：5 表设计（metadata_versions / catalog_tables / catalog_columns / import_jobs / import_errors）合理，metadata_version 作为逻辑分片键的设计符合"多版本元数据共存"的数仓需求
2. **WAL + 外键**：SQLite WAL 模式 + foreign_keys=ON 是正确选择，提升并发读写能力
3. **大小写不敏感**：normalized_table_name / normalized_column_name 双字段设计，原名称+标准化名称分离，兼顾可读性和查询便利，符合 Hive/Spark 数仓习惯
4. **前端六态 PageMode**：empty → ready → analyzing → analyzed → dirty → failed 的状态机设计覆盖了数据开发日常反复修改 SQL → 重新分析的完整交互循环
5. **蛇形命名一致性**：Python Pydantic / TypeScript / API JSON 全部使用 snake_case，避免驼峰转换混淆，适合后端主导的数据工具
6. **稳定 ID 规则**：`table:catalog.schema.table` 格式的实体 ID 可读、可追溯，便于排查血缘链路上的具体节点

#### 需改进项（详见下一节）

### 不适合数据开发日常使用的问题

#### 问题 1（中）: `metadata_versions.table_count` 字段是死字段
- **文件**: `backend/app/db/migrations/001_metadata.sql` 第 10 行
- **问题**: DDL 定义了 `table_count INTEGER DEFAULT 0`，但 `MetadataRepository` 的 `create_metadata_version` 方法从未写入该字段，`upsert_table` 也未更新它。该字段永远为 0。
- **影响**: 元数据版本列表页无法展示真正的表数量，属于元数据不完整。
- **建议**: 在 `upsert_table` 后更新对应版本的 `table_count`（通过 `UPDATE metadata_versions SET table_count = (SELECT COUNT(*) FROM catalog_tables WHERE metadata_version=?) WHERE version=?`），或者在 `get_metadata_context` 中动态计算。

#### 问题 2（中）: `catalog_tables.schema_name` 与导入模型中 `MetadataTableInput.schema` 字段命名不一致
- **文件**: `backend/app/db/migrations/001_metadata.sql` 第 17 行 vs `backend/app/domain/contracts.py` 第 165 行
- **问题**: 数据库列名叫 `schema_name`，但 Pydantic 导入模型叫 `schema`。虽然前端 contracts.ts 也一致地用 `schema`，但这种不一致可能导致 API 到数据库的映射出错或 reviewer 困惑。
- **建议**: 统一命名或在模型字段上添加 `alias="schema_name"` 映射。

#### 问题 3（低）: 目前没有对外的元数据导入 API 端点
- **文件**: `backend/app/main.py` 第 67 行
- **问题**: 只注册了 `health_router`，`MetadataImportRequest` / `MetadataImportResult` 的 Pydantic 模型已定义，但 `/api/metadata/import` 端点未实现。frontend 中的 Import 按钮是 stub。
- **影响**: P0 核心链路"JSON 元数据导入 → SQLite 元数据可查询"中的第一步缺少对外接口，只能通过 Python 脚本直接调 repository。
- **建议**: R02 优先实现 `/api/metadata/import` 端点。

#### 问题 4（中）: `get_table_by_name` 对输入自动 lowercase，但未在契约文档中声明
- **文件**: `backend/app/repositories/metadata_repository.py` 第 210 行（`table_name.lower()`）
- **问题**: `get_table_by_name` 调用时做了 `table_name.lower()`，但传入 `get_metadata_version` 后也做了 lowercase lookup（这没错），问题在于如果 `metadata_version` 参数本身带大小写，`get_metadata_version` 走的是精确匹配而非忽略大小写。只有 "latest" 这个魔法值有特殊处理。
- **建议**: 在接口文档中明确声明 lookup 为大小写不敏感。

#### 问题 5（低）: `import_jobs` 和 `import_errors` 两张表在 repository 中没有对应的 CRUD 方法
- **文件**: `backend/app/repositories/metadata_repository.py`
- **问题**: DDL 中已经定义了 `import_jobs` 和 `import_errors` 两张表，但 MetadataRepository 没有提供任何操作这两张表的方法（create_import_job / record_import_error 等）。
- **影响**: 导入过程中无法记录 job 状态和错误详情，影响问题排查。
- **建议**: R02 实现 `/api/metadata/import` 时补充对应的 repository 方法。

### 功能冗余问题

#### 问题 6（低）: `AnalysisOptions.include_expression_lineage` 字段在 P0 阶段无意义
- **文件**: `backend/app/domain/contracts.py` 第 269 行, `frontend/src/types/contracts.ts` 第 206 行
- **问题**: P0 明确不做表达式级血缘，但 AnalysisOptions 中包含了 `include_expression_lineage: boolean`。当前只能发送 `false`，且后端没有对应的表达式解析逻辑。
- **判断**: 属于预留字段，不算严格冗余，但建议在该字段上标注 `@deprecated_in_p0` 注释或标记为 `always_false_in_p0`，避免未来开发者误用。

#### 问题 7（低）: `SemanticsReport` 中的 7 个子字段全为空壳
- **文件**: `backend/app/domain/contracts.py` 第 321-329 行
- **问题**: `result_grain` / `filters` / `metrics` / `joins` / `windows` / `dedup_logic` / `semantic_risks` 共 7 个字段，P0 全部为空。虽然在 P0 规格中已声明为预留，但 7 个字段占 AnalysisResult 体积较大。
- **判断**: 预留合理，不视为严格冗余。但建议在 P1 启动语义分析前保持空壳。

#### 整体冗余评估
**R00+R01 范围内无明显冗余功能。** 契约类型中的预留字段（SemanticsReport 空壳、expression 相关类型、multi-table-union 预留）均已在 P0 规格说明书中声明为"预留但不要求实现"，符合预先设计。
