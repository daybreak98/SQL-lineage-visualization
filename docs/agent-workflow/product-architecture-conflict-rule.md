# 产品与架构冲突处理规则

## 概述

本文档定义了 senior-product-manager 和 senior-architect 意见冲突时的处理规则。

---

## 冲突处理原则

### 优先采用产品经理方案的条件

当同时满足以下条件时，优先采用产品经理的产品方案：

1. **技术可实现**
   - 技术上没有根本性障碍
   - 有可行的技术路径

2. **性能有保障**
   - 不会引入严重的性能问题
   - 响应时间在可接受范围内

3. **稳定性有保障**
   - 不会引入稳定性风险
   - 有基本的异常处理

4. **复杂度可控**
   - 不会引入过度复杂的设计
   - 开发和维护成本合理

5. **可维护性有保障**
   - 代码结构清晰
   - 便于后续维护

6. **不破坏核心架构边界**
   - 不违反模块化设计
   - 不破坏高内聚低耦合原则

7. **不引入明显过度耦合**
   - 模块间依赖合理
   - 没有隐藏依赖

---

## 架构师必须给出的评估

如果产品经理方案存在明显技术风险，senior-architect 必须给出：

### 1. 风险说明
- 明确指出技术风险点
- 说明风险的影响范围
- 评估风险发生的概率

### 2. 可替代方案
- 提供技术上更可行的方案
- 说明替代方案的优缺点
- 评估替代方案的开发成本

### 3. 最小可行实现
- 提供最小可行的实现方案
- 满足核心需求
- 降低技术风险

### 4. 是否建议进入下一轮迭代
- 建议立即实现
- 建议推迟到下一轮
- 建议进一步评估

### 5. 是否需要产品经理二次决策
- 是否需要产品经理重新评估
- 是否需要更多信息

---

## 冲突处理流程

```
senior-product-manager 提出产品方案
  ↓
senior-architect 评估技术可行性
  ↓
技术可行？
  ├─ 是 → 采用产品经理方案
  └─ 否 → 架构师给出风险评估
    ↓
  风险评估
    ├─ 风险说明
    ├─ 可替代方案
    ├─ 最小可行实现
    ├─ 进入下一轮建议
    └─ 二次决策需求
    ↓
  senior-product-manager 二次决策
    ├─ 接受替代方案 → 采用替代方案
    ├─ 坚持原方案 → 采用最小可行实现
    └─ 推迟到下一轮 → 归档
```

## 冲突处理示例

### 示例 1：实时血缘分析

**产品经理方案**：实时分析 SQL 血缘，输入即时反馈

**架构师评估**：
- 风险：复杂 SQL 解析可能耗时较长，影响用户体验
- 替代方案：异步分析 + 进度提示
- 最小可行实现：简单 SQL 实时分析，复杂 SQL 异步分析
- 建议：进入下一轮迭代
- 二次决策：需要产品经理确认异步方案

**最终决策**：采用最小可行实现，简单 SQL 实时分析，复杂 SQL 异步分析

---

### 示例 2：血缘画布拖拽

**产品经理方案**：血缘画布支持自由拖拽，节点可任意移动

**架构师评估**：
- 风险：无
- 评估：技术可行，性能有保障

**最终决策**：采用产品经理方案

---

### 示例 3：跨库血缘

**产品经理方案**：支持跨数据库的血缘分析

**架构师评估**：
- 风险：不同数据库 SQL 语法差异大，解析复杂度高
- 替代方案：先支持主流数据库（MySQL、PostgreSQL、Hive）
- 最小可行实现：支持 MySQL 和 Hive 的血缘分析
- 建议：分阶段实现
- 二次决策：需要产品经理确认优先支持的数据库

**最终决策**：先支持 MySQL 和 Hive，后续迭代扩展

---

## 注意事项

1. **沟通优先**：冲突应通过充分沟通解决，避免对抗
2. **数据支撑**：技术风险评估应有数据支撑
3. **用户导向**：最终决策应以用户价值为导向
4. **迭代思维**：复杂功能可以分阶段实现
5. **文档记录**：冲突处理过程和结果应记录在文档中
