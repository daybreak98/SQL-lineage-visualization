# 测试门禁规则

## 概述

本文档定义了 SQL 血缘解析可视化项目的测试门禁规则，确保每个功能点的质量。

---

## 核心规则

### 1. 每个功能点必须有测试

- 后端功能必须有单元测试
- 前端功能必须有组件测试或集成测试
- API 接口必须有接口测试

### 2. 后端功能必须由 senior-test-engineer 测试

- 后端开发完成后，必须交由测试开发验证
- 测试开发独立执行测试，不受开发 Agent 干扰

### 3. 测试 case 必须 100% 通过

- 任何测试 case 失败都必须修复
- 不允许跳过测试（除非有明确的临时豁免）

### 4. 不通过不得进入下一轮

- 测试未通过的功能点不得进入下一轮迭代
- 必须修复所有失败 case 后才能继续

---

## 测试开发检查项

### 功能正确性

- 功能是否按需求文档实现
- 输出结果是否正确
- 边界条件是否处理

### API 契约

- 请求格式是否符合接口契约
- 响应格式是否符合接口契约
- 错误码是否正确
- HTTP 状态码是否正确

### 回归风险

- 是否影响已有功能
- 是否有破坏性变更
- 是否需要回归测试

### 过度耦合

- 模块间依赖是否合理
- 是否有隐藏依赖
- 是否违反模块边界

### 异常处理

- 异常情况是否处理
- 错误提示是否清晰
- 是否有未捕获的异常

### 边界 Case

- 空输入处理
- 超大输入处理
- 特殊字符处理
- 并发情况处理

---

## 测试流程

```
开发 Agent 完成功能
  ↓
senior-test-engineer 接收测试任务
  ↓
执行功能测试
  ↓
执行回归测试
  ↓
执行接口测试
  ↓
执行边界测试
  ↓
测试结果
  ├─ 100% 通过 → 出具测试报告，允许进入下一轮
  └─ 有失败 → 输出失败 case，返回开发 Agent 修复
```

---

## 测试报告格式

```markdown
## 测试报告

### 测试范围
[范围说明]

### 测试 case 清单
- [case1]: 描述
- [case2]: 描述

### 通过 case
- [case1]: 通过
- [case2]: 通过

### 失败 case
- [case3]: 失败原因
  - 修复建议：[建议]

### 回归风险
[风险评估]

### 耦合风险
[风险评估]

### 是否允许进入下一轮
是/否

### 最终结论
通过 / 不通过
```

## 测试框架

### 后端测试

- 框架：pytest
- 运行命令：`pytest tests/`
- 覆盖率：`pytest --cov=src tests/`

### 前端测试

- 框架：Jest / Vitest
- 运行命令：`npm run test` 或 `yarn test`

### API 测试

- 框架：pytest + httpx
- 运行命令：`pytest tests/api/`
