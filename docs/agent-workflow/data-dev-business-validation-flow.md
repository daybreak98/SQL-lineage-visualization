# 数据开发业务验证规则

## 概述

本文档定义了 SQL 血缘解析可视化项目的数据开发业务验证规则，确保产品符合数据开发日常使用需求。

---

## 核心规则

### 1. 项目初期由数据开发筛选和加工测试用例

- senior-data-developer 在项目初期负责
- 筛选贴近真实业务的 SQL case
- 加工测试用例，覆盖核心场景

### 2. 每轮功能迭代后由数据开发做业务可用性验证

- 每轮功能点完成后触发
- senior-data-developer 进行业务测试
- 验证功能是否适合数据开发日常使用

### 3. 数据开发重点检查

#### 是否适合日常数据开发使用
- 操作是否便捷
- 链路是否过长
- 字段解释是否清晰

#### SQL 场景是否真实
- SQL 是否贴近真实业务
- 是否覆盖常见 SQL 模式

#### 元数据 case 是否完整
- 表结构信息是否完整
- 字段注释是否准确

#### 功能是否冗余
- 是否有不必要的功能
- 是否有重复功能

#### 操作链路是否过长
- 核心操作步骤数
- 是否有优化空间

#### 字段解释是否清晰
- 字段含义是否明确
- 是否有歧义

#### 血缘展示是否符合数据开发排查习惯
- 展示方式是否直观
- 是否便于问题排查

### 4. 问题提交给产品经理

- senior-data-developer 输出业务测试反馈文档
- 提交给 senior-product-manager

### 5. 产品经理决定是否进入下一轮迭代文档

- senior-product-manager 评估反馈
- 决定是否纳入下一轮迭代

---

## 业务验证流程图

```
功能点开发完成
  ↓
senior-test-engineer 测试通过
  ↓
project-orchestrator 调度业务验证
  ↓
senior-data-developer 进行业务测试
  ↓
检查项
  ├─ 日常使用适配性
  ├─ SQL 场景真实性
  ├─ 元数据完整性
  ├─ 功能冗余
  ├─ 操作链路
  ├─ 字段解释清晰度
  └─ 血缘展示习惯
  ↓
输出业务测试反馈文档
  ↓
senior-product-manager 收到反馈
  ↓
产品经理决策
  ├─ 纳入下一轮迭代 → 写入迭代文档
  └─ 不纳入 → 归档保存
```

## 测试用例来源

### 项目现有测试用例

- `测试用例/00_血缘解析基准/`
- `测试用例/数仓/`
- `测试用例/搜索/`
- `测试用例/AB实验/`
- `测试用例/优惠先享需求/`
- `测试用例/sql血缘/`

### 需要覆盖的 SQL 场景

1. **基础查询**
   - SELECT 单表
   - WHERE 条件
   - GROUP BY 聚合
   - ORDER BY 排序

2. **多表关联**
   - INNER JOIN
   - LEFT/RIGHT JOIN
   - CROSS JOIN
   - 自连接

3. **子查询**
   - 标量子查询
   - 行子查询
   - 表子查询
   - 关联子查询

4. **CTE (WITH 语句)**
   - 简单 CTE
   - 递归 CTE
   - 多 CTE

5. **高级特性**
   - 窗口函数
   - UNION / UNION ALL
   - INTERSECT / EXCEPT
   - CASE WHEN

6. **血缘场景**
   - 表级血缘
   - 字段级血缘
   - 跨库血缘
   - 视图血缘
   - 嵌套血缘
