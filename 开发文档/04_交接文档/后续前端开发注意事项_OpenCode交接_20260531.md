后端：

```bash
cd backend
python -m uvicorn app.main:app --host 127.0.0.1 --port 8000
```

前端：

```bash
cd frontend
npm run dev
```

访问：

```text
http://localhost:5173
```



## 三、API 接入规则

### 统一入口

所有前端请求必须优先写在：

```text
frontend/src/api/client.ts
```

不要在组件中散落 `fetch('/api/...')`。

### 当前已封装 API

```ts
getHealth()
formatSql(sql, dialect)
analyzeSql(sql, dialect)
previewMetadata(payload)
commitMetadata(payload)
listMetadataTables()
listMetadataColumns(table)
```

### 新增后端接口时的处理顺序

1. 在 `frontend/src/types/lineage.ts` 或更合适的类型文件中定义响应类型。
2. 在 `frontend/src/api/client.ts` 增加函数。
3. 在组件中调用 API client。
4. 增加 loading / error / empty / success 状态。
5. 跑 `npm run build`。
6. 如果是用户交互能力，补组件测试或手工验证记录。

## 四、Analyze 后续开发注意事项

当前 `/api/sql/analyze` 后端仍是 stub。

现状：

```text
后端返回 tables_extracted / columns_extracted
前端转换为 backendGraph / backendSearchItems
```

后续 R04a 正式实现后，前端应优先消费：

```text
graph_view_model.nodes
graph_view_model.edges
diagnostics_report.diagnostics
source_locations
metadata_context
```

不要长期依赖：

```text
tables_extracted
columns_extracted
```

这些只是当前 stub 兼容字段。

### Analyze 适配优先级

1. `AnalysisResult.status`
2. `diagnostics_report.diagnostics`
3. `graph_view_model.nodes`
4. `graph_view_model.edges`
5. `source_locations`
6. `metadata_context`
7. `semantics_report`

### 不允许

- 不允许前端自行推断字段血缘作为最终结果。
- 不允许把 mockLineage 的边当成后端真实血缘。
- 不允许在后端返回 failed 时仍显示 trusted。
- 不允许吞掉 API 错误后静默显示成功。

## 七、状态模型注意事项

当前核心状态在：

```text
frontend/src/types/lineage.ts
WorkbenchState
```

新增的后端适配字段：

```ts
backendGraph
backendSearchItems
backendDiagnostics
backendMessage
metadataLabel
```

这些字段的意义：

| 字段 | 用途 |
|---|---|
| `backendGraph` | 后端分析结果转换后的画布节点和边 |
| `backendSearchItems` | 后端分析结果转换后的搜索候选 |
| `backendDiagnostics` | 后端 diagnostics 转换后的前端诊断 |
| `backendMessage` | 最近一次后端请求状态说明 |
| `metadataLabel` | 顶部 metadata badge 显示 |

### 状态语义

| 状态 | 含义 |
|---|---|
| `pageMode=ready` | 有 SQL，但未分析 |
| `pageMode=analyzing` | 正在调用后端 |
| `pageMode=analyzed` | 后端分析成功或 partial |
| `pageMode=failed` | 后端调用失败或分析失败 |
| `trustStatus=trusted` | 当前图谱来自最新分析结果 |
| `trustStatus=stale` | SQL 已改动，图谱过期 |
| `trustStatus=untrusted` | 未分析或失败 |

不要只改 UI 文案而不更新状态。

## 八、Mock 数据使用规则

仍保留：

```text
frontend/src/data/mockLineage.ts
```

允许用途：

- 默认示例展示
- 后端未返回某类能力前的视觉兜底
- UI 结构开发时的占位

禁止用途：

- 新增业务能力时优先写 mock。
- 用 mock 结果覆盖后端真实结果。
- 后端失败时伪装成成功。
- 将 mockLineage 扩展成第二套业务状态源。

开发判断原则：

```text
能从后端拿到，就不要从 mock 拿。
后端没有提供，就明确标记为 fallback / placeholder。
```

## 九、UI 结构注意事项

当前 v1.4 UI 分层：

```text
App
├── TopBar
├── LeftNav
├── SqlEditorPanel
├── Splitter
├── SearchBar
├── CanvasToolbar
├── LineageCanvas
├── DetailPanel
├── StatusStrip
├── Drawer
└── MetadataDialog
```

### 推荐改动位置

| 需求 | 文件 |
|---|---|
| 顶部按钮、metadata badge | `TopBar.tsx` |
| SQL 编辑器 | `SqlEditorPanel.tsx` |
| 搜索结果 | `SearchBar.tsx` |
| 画布节点边 | `LineageCanvas.tsx` + `data/selectors.ts` |
| 底部详情 | `DetailPanel.tsx` |
| 诊断抽屉 | `Drawer.tsx` |
| 元数据导入 | `MetadataDialog.tsx` |
| 后端请求 | `api/client.ts` |
| 类型 | `types/lineage.ts` |

### 不建议

- 不建议把大量业务逻辑塞进 `TopBar`。
- 不建议在 `LineageCanvas` 里直接请求后端。
- 不建议在 `SearchBar` 中重新解析 SQL。
- 不建议重新引入 Zustand，除非有明确架构决策。

## 十一、已知技术债

| 技术债 | 说明 | 建议 |
|---|---|---|
| Analyze 后端仍是 stub | 当前只返回 parse 级信息 | 等 R04a 完成后切正式 AnalysisResult |
| 前端测试缺失 | v1.4 迁移后没有恢复 Vitest | 下一轮补 API client + 组件测试 |
| mockLineage 仍较重 | 仍承载默认图、详情、路径 | 随后端能力逐步替换 |
| Browser 插件不可用 | 当前环境截图/交互自动化失败 | 可用 Playwright 或修复 Browser 插件环境 |
| Metadata 弹窗基础版 | 已可用，但 UX 仍粗 | 后续 UI hardening |

## 十二、给 OpenCode 的最小继续任务建议

如果继续开发，建议按以下顺序：

1. 为 `frontend/src/api/client.ts` 补测试。
2. 为 `MetadataDialog` 增加上传 JSON 文件能力。
3. 将 Analyze 的 `graph_view_model` 映射逻辑拆成独立 mapper。
4. 等后端 R04a 完成后，切正式 `AnalysisResult`。
5. 清理或降级 `mockLineage.ts` 中被后端覆盖的部分。
6. 增加前后端联调测试脚本。

---

## 十三、红线

后续前端开发不要做以下事情：

1. 不要修改 `sql-lineage-workbench-v1.4/` 作为主线。
2. 不要恢复旧的单文件 `frontend/index.html` 实现。
3. 不要绕过 `frontend/src/api/client.ts` 直接散写 fetch。
4. 不要把 mock 结果当后端结果。
5. 不要在后端 failed/offline 时显示 trusted。
6. 不要让前端自行计算最终字段血缘。
7. 不要破坏 `/api` 代理配置。

---

## 十四、最终交接结论

当前前端已经是 `frontend/` 下的 React v1.4 主线，并已接入后端当前可用服务。

后续 OpenCode 应继续在 `frontend/` 内开发，并把 `frontend/src/api/client.ts` 作为所有后端能力接入的唯一入口。
