# v0.6 需求文档的 Agent 开发就绪度评估

## 总体判断：当前就绪度约 45%，距离可直接交给 agent 开发还差约 55% 的补充工作。

---

## 一、agent 开发需要什么而 v0.6 缺少什么

### 按 agent 执行 R00（项目初始化）举例

v0.6 写的是：

> 初始化 React + TypeScript + Vite
> 建立 Workbench 页面骨架

agent 收到这个指令后会做什么？它会创建一个默认的 Vite + React 项目，放一个空白 `Workbench.tsx`。但你知道你想要的是什么吗？——不知道。因为 v0.6 没有说：

| 缺失信息 | agent 会怎么做 | 你需要什么 |
|---|---|---|
| React 版本 | 用最新版 | 未知 |
| UI 组件库是什么 | 可能不用任何库 | 可能想用 Ant Design / MUI / shadcn |
| Workbench 布局的具体分栏比例 | 随便做 | 可能想要经典的 3 栏或 4 栏 |
| 状态管理用 Zustand 的 store 结构 | 不会创建 store | v0.6 提到了 Zustand 但没定义 store shape |
| 路由策略 | 可能用 react-router | 可能单页就够了 |
| ESLint/Prettier 配置 | 默认 | 可能团队有规范 |

**一个需求从 v0.6 的"做什么"到 agent 的"怎么做"，中间缺失了大量决策和约束。**

## 二、逐层缺失清单

### 2.1 缺失层 1：技术栈版本锁定与初始化配置（影响 R00）

| 缺失项 | 严重程度 | 说明 |
|---|---|---|
| 各依赖的精确版本或版本范围 | 高 | React 18 vs 19、FastAPI 0.100+ vs 0.115+ 差异很大 |
| 项目构建工具配置 | 高 | Vite 配置、FastAPI 启动方式（uvicorn/gunicorn） |
| UI 组件库选型 | 中 | 不指定则 agent 可能裸写 CSS 或选你不喜欢的库 |
| 包管理器 | 低 | npm / pnpm / yarn |
| CSS 方案 | 中 | Tailwind / CSS Modules / styled-components |
| TypeScript 严格模式配置 | 中 | strict: true 还是宽松 |
| 代码格式化/检查工具 | 中 | ESLint + Prettier 规则 |

### 2.2 缺失层 2：数据库 DDL（影响 R01）

v0.6 写的是：

> `catalog_tables` | 表基础信息
> `catalog_columns` | 字段名、类型、注释、分区标识

agent 需要的是：

```sql
CREATE TABLE IF NOT EXISTS catalog_tables (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    catalog_name TEXT NOT NULL DEFAULT 'default',
    schema_name TEXT NOT NULL DEFAULT 'default',
    table_name TEXT NOT NULL,
    table_comment TEXT,
    table_type TEXT DEFAULT 'table',
    created_at TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at TEXT NOT NULL DEFAULT (datetime('now')),
    UNIQUE(catalog_name, schema_name, table_name)
);
```

| 缺失项 | 严重程度 | 说明 |
|---|---|---|
| 所有表的完整 DDL | **致命** | 没有 DDL，agent 无法创建数据库 |
| 字段类型、默认值、约束 | **致命** | TEXT vs INTEGER、NOT NULL、UNIQUE 等 |
| 索引策略 | 高 | 哪些字段需要索引以支撑补全和搜索 |
| 迁移策略 | 高 | Alembic？还是直接执行 SQL 文件？ |
| 外键关系 | 中 | import_batches 和 import_changes 之间的关联 |
| 时间戳策略 | 中 | 用 TEXT 还是 INTEGER 存储时间 |

### 2.3 缺失层 3：API 接口完整规格（影响 R04a）

v0.6 给出了请求/响应结构骨架，但对 agent 来说还不够：

| 缺失项 | 严重程度 | 说明 |
|---|---|---|
| 每个字段的必填/可选/默认值标注 | 高 | agent 需要知道哪些字段可以为 null |
| 错误响应格式 | 高 | 当分析失败时返回什么结构？HTTP 状态码？ |
| 所有 API 的完整列表和路由 | 高 | v0.6 只详细定义了 /api/sql/analyze，缺少导入/搜索/hover/completion 等 API |
| 每个 API 的 curl 示例 | 中 | agent 做集成测试需要 |
| 认证/鉴权策略 | 低 | P0 是否完全无鉴权？ |
| CORS 配置 | 中 | 前后端分离开发时的跨域策略 |

**v0.6 缺少的后端 API 规格：**

| API | v0.6 覆盖程度 |
|---|---|
| POST /api/sql/analyze | 详细 ✓ |
| POST /api/sql/format | 仅提及 |
| POST /api/metadata/import/preview | 仅提及 |
| POST /api/metadata/import/commit | 仅提及 |
| GET /api/metadata/tables | 未定义 |
| GET /api/metadata/tables/{id}/columns | 未定义 |
| POST /api/editor/completion | 未定义 |
| POST /api/editor/hover | 未定义 |

### 2.4 缺失层 4：核心数据模型的 Python/TypeScript 类型定义（影响 R05-R07）

v0.6 给了 JSON 示例，但 agent 需要的是代码级类型定义：

| 缺失项 | 严重程度 | 说明 |
|---|---|---|
| Python Pydantic 模型定义 | **致命** | AnalysisResult、LineageIR、SourceLocation 等的字段名、类型、验证规则 |
| TypeScript interface/type 定义 | **致命** | 前端对应的类型，agent 可能用不同的命名和结构 |
| 枚举值的代码定义 | 高 | DiagnosticCode 的 Python Enum 和 TS enum |
| 字段命名规范 | 中 | Python 用 snake_case，TS 用 camelCase，中间如何转换 |

举例——v0.6 的 SourceLocation JSON：

```json
{ "start_line": 3, "start_col": 8, "end_line": 3, "end_col": 16 }
```

agent 可以写出多种 Pydantic 模型，但你需要的是哪种？

```python
# 方案 A：统一字段名
class SourceLocation(BaseModel):
    start_line: int
    start_col: int

# 方案 B：用 alias 适配 Python 命名
class SourceLocation(BaseModel):
    start_line: int = Field(alias="startLine")
    start_col: int = Field(alias="startCol")
```

这些选择直接影响前后端数据交互，v0.6 没有给出约定。

### 2.5 缺失层 5：按需求的文件级实现规格

v0.6 写的是：

> ScopeResolver | 识别主查询、表别名、简单 from 表

agent 需要知道：

| 缺失项 | 严重程度 | 说明 |
|---|---|---|
| 该文件的公开函数签名 | 高 | `resolve(ast) -> ScopeResult` 还是什么？ |
| 输入输出类型 | **致命** | 输入 SQLGlot 的什么对象？输出什么结构？ |
| 异常处理策略 | 中 | 解析失败时抛异常还是返回错误对象？ |
| 依赖的其他模块 | 高 | ScopeResolver 依赖哪些模块？谁创建它？ |
| 单元测试的最小覆盖用例 | 中 | 至少 3-5 个具体测试用例及预期结果 |

### 2.6 缺失层 6：前端组件规格

| 缺失项 | 严重程度 | 说明 |
|---|---|---|
| 每个组件的 Props 接口 | **致命** | SqlEditor 需要哪些 props？onAnalyze 回调？ |
| 组件的状态管理边界 | 高 | 哪些状态放 Zustand store，哪些放组件本地 state |
| Monaco Editor 的具体配置 | 高 | 主题、语言注册、completion provider 注册方式 |
| React Flow 的节点自定义组件规格 | 高 | 每种 node_type 对应的 React 组件长什么样 |
| 组件间通信方式 | 中 | props drilling / context / store / event bus |
| 加载/空/错误状态的 UI 规格 | 中 | 每种状态下的展示要求 |

### 2.7 缺失层 7：具体到可执行的 P0 验收测试用例

v0.6 写的是：

> 单表字段解析：`select order_no from t` 可解析来源字段

agent 需要：

| 缺失项 | 严重程度 | 说明 |
|---|---|---|
| 完整的测试 SQL 文本 | 高 | 不是 "select order_no from t"，而是带元数据上下文的完整测试 |
| 预期的 JSON 输出 | **致命** | 断言什么？LineageIR.nodes 应该有哪些？顺序重要吗？ |
| 测试元数据 fixture | **致命** | 测试前需要向 SQLite 插入什么元数据？ |
| 边界用例 | 中 | 空 SQL、超长 SQL、特殊字符字段名等 |

### 2.8 缺失层 8：开发顺序内的任务依赖细节

v0.6 给出了需求间的宏观依赖（R00→R01→R02...），但缺少：

| 缺失项 | 严重程度 | 说明 |
|---|---|---|
| 一个需求内子任务的执行顺序 | 高 | R00 的前端和后端工作可以并行，但 v0.6 没标 |
| 文件间的创建依赖 | 高 | domain/scope_model.py 必须在 services/scope_resolver.py 之前创建 |
| 每个任务预估的 agent 执行步数 | 低 | 用于判断是否应该拆得更细 |

## 三、具体补充方案

### 优先级分类

#### 致命级（不做则 agent 无法正确开始）

1. **数据库 DDL 脚本**：9 张核心表的完整 CREATE TABLE 语句
2. **Pydantic 模型定义**：所有 domain 对象的 Python 类型
3. **TypeScript 类型定义**：所有前端数据模型的 TS interface
4. **完整 API 规格**：每个 API 的 method/path/request body/response body/error format
5. **P0 测试用例的输入/预期输出**：至少 5-8 个 Golden Case 的完整 fixture + 断言

#### 高级（做了才能保证 agent 不偏离设计意图）

6. **技术栈版本清单**：所有依赖的精确版本
7. **Monaco Editor 配置规格**：主题、语言、provider 注册方式
8. **React Flow 节点/边自定义组件规格**：每种类型的外观和行为描述
9. **每个 service 的公开接口签名**：类名、方法名、参数类型、返回类型
10. **状态管理 store 的 shape 定义**：每个 Zustand store 的初始状态和 action

#### 中级（提升 agent 输出质量和一致性）

11. **代码风格规范**：命名约定、文件组织方式、注释策略
12. **错误处理协议**：后端统一的异常类和错误响应格式
13. **日志规范**：用什么 logger、什么级别记录什么
14. **前端路由表**：所有页面路由及其对应的组件
15. **环境变量清单**：DATABASE_PATH、CORS_ORIGINS 等

## 六、结论

v0.6 是一份优秀的**人类开发者**需求文档——架构清晰、依赖正确、范围可控。但它不是一份 **agent 开发规格书**。差距的核心不在于"信息不够多"，而在于"信息形态不对"——v0.6 的信息是**描述性的**（"识别主查询、表别名"），agent 需要的是**声明性的**（"输入 SQLGlot exp.Select，输出 ScopeResult，其中包含..."）和**结构化的**（"CREATE TABLE catalog_tables (...);"）。

**建议优先补充致命级的 5 项（DDL、Pydantic 模型、TS 类型、API 完整规格、Golden Case 测试），完成后就绪度可提升至约 75%，agent 即可开始 P0 的实质性开发工作。**
