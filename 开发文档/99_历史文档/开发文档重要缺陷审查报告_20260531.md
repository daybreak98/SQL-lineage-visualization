# 开发文档重要缺陷审查报告

> 审查日期：2026-05-31  
> 审查范围：项目目标、主需求文档、Agent 模块拆解文档、阶段开发规格、前端页面文档、历史归档与决策、测试门禁清单、当前代码目录结构。  
> 审查目标：识别会影响后续 Agent 迭代、实现顺序、接口契约、测试门禁和前后端联调的重要文档缺陷，并给出修复建议。  
> 约束说明：本次只新增审查建议文档，不修改业务代码，不覆盖现有文档。

---

## 一、总体结论

当前文档体系已经具备较完整的产品目标、阶段规划、模块拆解、测试门禁和多 Agent 协作规则，整体方向是清晰的。

但存在 7 个重要缺陷，其中 5 个会直接影响下一轮开发执行：

1. 当前迭代状态源不一致，R02 在测试清单中已完成，但历史输入仍指向 R02。
2. 主需求文档与模块拆解路线图的 P0 执行顺序不一致。
3. 前端权威工程入口不清晰，`frontend/` 与 `sql-lineage-workbench-v1.4/` 同时存在且文档引用漂移。
4. `/api/sql/analyze` 契约与当前阶段实现边界不清，容易让前端基于临时 stub 固化逻辑。
5. 测试门禁的“通过”表述过宽，未区分 R02 后端通过、前端未跑、Golden Case 未完成、API Contract Test 未完成。
6. `schema` / `schema_name` 的外部 API、内部模型、数据库字段映射规则没有形成唯一权威说明。
7. 契约模型文件组织在不同文档中存在“单文件 contracts.py”与“拆分领域模型文件”的冲突。

建议先做一次“文档基线治理”再进入下一轮功能开发，否则后续 Agent 可能重复实现已完成模块、修改错误前端目录，或绕过未来的 ContractAssembler 与 Golden Case 门禁。

## 三、重要缺陷与建议

### D1：当前迭代状态源不一致

**严重级别：阻断级**

`tests/p0_test_checklist.md` 已声明 `R02 ✅ 全部通过（31/31）`，并列出了 JSON 元数据导入的 16 个集成测试通过。但 `开发文档/99_历史文档/下一轮迭代输入_R02_20260530.md` 仍然把 R02 作为下一轮输入，`迭代归档_R00R01_20260530.md` 也只归档到 R00 + R01。

这会导致 project-orchestrator 在读取历史输入时误判当前进度，重复分派 R02，或者把已经完成的 `table_count`、`import_jobs`、`schema_name` 修复再次作为待办。

**建议：**

1. 新增 `开发文档/99_历史文档/迭代归档_R02_20260531.md`，明确 R02 的实现文件、测试结果、未完成项。
2. 新增或更新一个当前迭代指针文档，例如 `开发文档/CURRENT_ITERATION.md`，只记录当前已完成到哪一轮、下一轮是什么。
3. 将 `下一轮迭代输入_R02_20260530.md` 标注为历史输入，新增 `下一轮迭代输入_R03_20260531.md` 或直接进入模块路线图定义的下一可执行模块。

---

### D2：主需求文档与模块拆解路线图的 P0 顺序不一致

**严重级别：阻断级**

主需求文档给出的 P0 顺序是：

```text
R00 → R01 → R02 → R03 → R04a → R04b → R08a → R09a → R05 → R06 → R07 → R08b/R08c → R17
```

模块拆解文档和 `MODULE_ROADMAP.md` 给出的推荐路径是：

```text
M01 → M02 → M03 → M04 → M06 → M05 → M07 → M08 → M09 → M10 → M11 → M12 → M13 → M14 → M15 → M16 → M17
```

两者核心差异包括：

- R04b 契约基线在模块拆解中提前到 M02，但主需求路径放在 R04a 后。
- Diagnostics primitives 在模块拆解中必须早于 R02，但主需求路径把 R08a 放在 R04b 后。
- R03 前端编辑器在主需求路径中早于 R04a/R04b，但模块拆解中 M15 依赖 M14 Analyze API。

这会让不同 Agent 按不同文档执行，产生实现顺序冲突。

**建议：**

1. 以 `开发文档/01_Agent模块拆解文档/README.md` 和 `MODULE_ROADMAP.md` 的 v1.1 路线作为当前唯一执行顺序。
2. 在主需求文档中追加“执行顺序已由模块拆解 v1.1 覆盖”的勘误说明。
3. 建立一张 R 编号到 M 编号的映射表，避免 R02/M05、R04b/M02、R04a/M14 这类编号跳跃造成误读。

---

### D3：前端权威工程入口不清晰

**严重级别：高风险**

多个文档把前端实现文件写成 `frontend/src/...`，例如 P0 阶段规格、R00+R01 归档和主需求文档。但当前工作区中，`frontend/` 目录主要剩下 `index.html`、`dist/`、`package.json` 等文件，而完整 React 源码位于 `sql-lineage-workbench-v1.4/src/`。

同时，`开发文档/03_前端页面文档/sql_lineage_workbench_v1_4_merged_final_notes.md` 记录了 v1.4 页面实现建议，但没有明确它是否已经替代 `frontend/src` 作为主工程。

这个缺陷会导致前端 Agent 修改错误目录，或者测试 Agent 按 `frontend/src` 查找不存在的 smoke test。

**建议：**

1. 明确当前前端权威目录：二选一。
   - 方案 A：将 `sql-lineage-workbench-v1.4/` 作为正式前端工程，并更新所有文档中的 `frontend/src` 路径。
   - 方案 B：将 v1.4 源码迁回 `frontend/src/`，保留 `frontend/` 作为唯一前端工程。
2. 在 `开发文档/README.md` 增加“当前前端入口”字段。
3. 在前端测试门禁中写清楚运行目录、命令和预期测试文件。

---

### D4：`/api/sql/analyze` 契约与临时实现边界不清

**严重级别：高风险**

主需求文档和 P0 阶段规格明确要求 `/api/sql/analyze` 返回唯一主模型 `AnalysisResult`。R04a 模块文档也要求通过 `AnalysisOrchestrator` 和 `ContractAssembler` 组装响应，并明确不得绕过 ContractAssembler 返回 dict。

当前代码中已有 `/api/sql/analyze` 端点，但它是 R04a 前的基础解析 stub，`response_model=dict`，返回 `tables_extracted`、`columns_extracted`、空 `graph_view_model` 等临时字段。

文档没有明确这个 stub 的生命周期、禁止前端固化依赖的规则，以及 R04a 交付时必须替换的清单。

**建议：**

1. 新增 `开发文档/01_Agent模块拆解文档/R04A_STUB_DEPRECATION_NOTE.md`，声明当前 `/api/sql/analyze` stub 只允许用于 R03 前端联调，不是正式契约。
2. 在 R04a 模块文档的验收标准中补充“删除或替换 stub 响应字段，response_model 改为 AnalysisResult”。
3. 前端文档明确：R04a 前只能做 UI 联调，不得把 `tables_extracted` / `columns_extracted` 作为最终业务字段。

---

### D5：测试门禁状态表述过宽

**严重级别：高风险**

`tests/p0_test_checklist.md` 标注 R02 全部通过，但同一文档仍保留以下未完成项：

- 前端 smoke test 仍是未执行状态。
- Golden Case `GC-P0-001 ~ GC-P0-008` 仍待实现或待统一回归。
- R02 HTTP 422 测试需要 API endpoint 实现后补充。
- API Contract Test 要等 R04a 实现。
- “后续模块不绕过 MetadataRepository 直接查 SQLite”仍需 code review 确认。

这不是 R02 后端不能进入下一步，而是“P0 总门禁尚未通过”。当前文档没有把模块门禁、阶段门禁、端到端门禁分开，容易让后续 Agent 误判质量状态。

**建议：**

1. 将测试门禁拆成三层：
   - 模块门禁：当前模块单测/集成测试是否通过。
   - 阶段门禁：P0 全部模块 Contract/Golden/前端/后端是否通过。
   - 发布门禁：前后端联调、真实业务 SQL、可视化验收是否通过。
2. 将 `R02 ✅ 全部通过（31/31）` 改写为“R02 后端模块测试通过；P0 阶段门禁未完成”。
3. 为 Golden Case 单独建立进度表，记录每个 case 的 input、metadata、expected、执行脚本和当前状态。

---

### D6：`schema` / `schema_name` 映射规则缺少唯一权威说明

**严重级别：中高风险**

文档和代码中同时出现：

- 外部 API 查询参数 `schema`
- 元数据 JSON 字段 `schema_name`
- SQLite 字段 `schema_name`
- Repository 方法参数 `schema`
- R02 遗留项“`schema` → `schema_name` 命名统一”

当前实现中 Repository 已有映射说明，但开发文档没有形成唯一的跨层契约。后续前端、导入服务、NameResolver、MetadataContext 很容易再次产生字段漂移。

**建议：**

1. 新增一节“Schema 命名分层约定”：
   - HTTP query 参数：`schema`
   - JSON payload 字段：`schema_name`
   - 数据库列名：`schema_name`
   - Python 内部方法参数：允许 `schema`，但必须显式映射到 `schema_name`
2. 将该约定写入 P0 阶段规格、R02 模块文档、R05 Metadata/NameResolver 模块文档。
3. 增加 Contract Test，覆盖 API 参数、JSON 导入字段和 Repository 查询三层一致性。

---

### D7：契约模型文件组织存在冲突

**严重级别：中风险**

R04b 模块文档把领域模型交付物拆成：

- `backend/app/domain/analysis_result.py`
- `backend/app/domain/stage_status.py`
- `backend/app/domain/diagnostics_model.py`
- `backend/app/domain/lineage_ir.py`
- `backend/app/domain/graph_view_model.py`
- `backend/app/domain/source_location.py`

而当前实现和 R00+R01 归档使用的是单文件：

- `backend/app/domain/contracts.py`

两种组织方式都可以，但文档没有明确“当前采用单文件，后续是否拆分、何时拆分、拆分是否属于重构”。这会导致后续 Agent 按 R04b 文档新建拆分文件，形成重复模型定义。

**建议：**

1. 当前阶段先冻结 `contracts.py` 为唯一 P0 契约入口。
2. 如果要拆分，应单独开一个“契约模型拆分重构”任务，并要求所有导入路径和 Contract Test 同步迁移。
3. 在 R04b 文档中追加说明：交付物文件名以当前仓库实际实现为准，禁止重复定义同名模型。

## 八、最终判断

项目不是缺少文档，而是已经进入“文档很多、但权威入口和当前状态需要收敛”的阶段。

如果不先修复上述缺陷，后续 Agent 最大风险不是写不出代码，而是按照错误入口写出“看似完成、实际偏航”的代码。优先建立当前迭代指针、唯一模块顺序、唯一前端入口、唯一契约边界，后续开发效率会明显提升。
