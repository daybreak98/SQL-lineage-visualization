## 三、数据开发反馈决策（7 个问题）

| # | 严重度 | 问题 | 决策 | 理由 |
|---|--------|------|------|------|
| 1 | 🟡中 | `metadata_versions.table_count` 死字段（DDL 定义但从不更新） | ✅ **进入 R02 修复** | 数据完整性 bug。R02 实现 import 链路时自然应修复——在 `upsert_table` 后更新 `table_count`。改动极小（1 行 UPDATE SQL），修复成本低。**验收标准**：导入完成后 `metadata_versions.table_count` 反映真实的 catalog_tables 数量。 |
| 2 | 🟡中 | `schema` vs `schema_name` 命名不一致（contracts.py `schema` vs 数据库列 `schema_name`） | ✅ **进入 R02 统一** | R02 实现导入端点时统一命名最为自然。建议在 Pydantic 模型字段上加 `alias="schema_name"` 映射，或在 migration 中将列名改为 `schema`。不改的话后续代码中两种命名交织，排查成本更高。**验收标准**：API 层、Pydantic 模型、数据库列命名关系在文档中有明确说明，无歧义。 |
| 3 | 🟢低 | `/api/metadata/import` 端点未实现 | ✅ **R02 核心任务** | 这是 R02 模块（M05）定义的核心交付物，不是额外决策项。R02 必须实现。 |
| 4 | 🟡中 | `get_table_by_name` 大小写契约未文档化 | ✅ **进入 R02 文档化** | 纯文档补充任务，成本极低。在 `get_table_by_name` 的 docstring 中声明 "table_name 参数大小写不敏感（内部执行 lowercase 归一化）"。**验收标准**：API 文档或 docstring 中明确声明大小写行为。 |
| 5 | 🟢低 | `import_jobs` / `import_errors` 表无 CRUD 方法 | ✅ **进入 R02 实现** | 与 `/api/metadata/import` 不可分割——导入时必须记录 job 状态和错误明细。R02 模块文档已约定此需求。**验收标准**：MetadataRepository 提供 `create_import_job` / `update_import_job` / `record_import_error` 方法，导入链路中调用。 |
| 6 | 🟢低 | `AnalysisOptions.include_expression_lineage` 字段 P0 无意义 | ⏸ **归档到 R03** | P0 规格明确声明为预留字段。R03 可在字段上添加 `@deprecated_in_p0` 注释。R02 与表达式无关，不处理。 |
| 7 | 🟢低 | `SemanticsReport` 7 个子字段为空壳 | 🚫 **不处理** | P0 规格明确声明 SemanticsReport 为预留空壳，符合设计。P1 启动语义分析前保持现状。不视为问题。 |

## 四、冗余功能决策

### 4.1 R00+R01 范围内冗余评估

**结论：无冗余功能。**

数据开发已验证：当前 R00+R01 实现的功能（WorkbenchShell、PageMode 六态、MetadataRepository 5 表基础 CRUD、Contract Baseline 枚举和模型）均为 P0 最小闭环的必要组成部分。

### 4.2 contracts.py 中 P1/P2 阶段预留枚举和模型

**决策：全部保留，不移除。**

保留的枚举和模型包括但不限于：

| 预留项 | P0 状态 | 保留理由 |
|--------|---------|---------|
| `AnalysisLevel.expression` | P2 才用 | 定义了契约基准，P1/P2 模块在此基础上扩展 |
| `GraphViewMode` 五种视图 | P0 仅 table/column | 前端 GraphViewModel 类型已依赖完整枚举 |
| `EntityType` 全部枚举值（scope/cte/subquery 等） | P0 部分使用 | 移除导致 TypeScript 类型漂移，后续回补成本高 |
| `SemanticsReport` 7 字段空壳 | P0 全部为空 | P0 规格明确允许预留空壳 |
| `SourceLocation` 完整字段 | P0 仅定义模型壳 | M07 将实现坐标转换工具，移除后需重新定义 |
| `LineageNodeType` / `LineageEdgeType` 非 P0 类型 | P0 部分使用 | 定义了完整的类型系统契约 |

**保留原则**：P0 开发规格 §1 明确声明"P0 可以预留但不要求完整实现"这些类型。移除预留类型会导致前后端类型漂移，修复成本远高于保留成本。

## 九、验收标准补充

### 9.1 R02 整体验收标准（产品侧）

1. `/api/metadata/import` 端点可用，支持 preview 和 commit 两种模式
2. 导入完成后的元数据可在 `/api/metadata/tables` 和 `/api/metadata/columns` 查询
3. 错误输入（非法 JSON、重复字段、空 columns）返回结构化 diagnostics，不自崩溃
4. 事务回滚生效：commit 过程出错后 catalog_tables/catalog_columns 保持不变
5. `import_jobs` 和 `import_errors` 表正确记录每次导入的状态和错误
6. `metadata_versions.table_count` 正确反映导入后的表数量
7. Empty 态 TopBar 有可点击的 "Load Example" 按钮
8. analyzed 态 Analyze 按钮视觉层级高于 Format/Import 按钮
9. 15 个已有测试无回归失败
10. 新增 Golden Case（metadata import 场景）通过

### 9.2 R02 不验收的内容（明确排除）

- 前端 Import 按钮联动后端 API（R02 仅实现后端，前端对接可在 R03 并行做）
- SQL 解析与血缘分析（R03+）
- 子查询/CTE Golden Case（R03+）
- 元数据 fixture 扩展到 5 表（R03+）
