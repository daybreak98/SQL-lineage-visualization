# SQL 血缘解析工作台｜六份技术文档评价报告

## 评价范围

对工作目录下的六份技术文档进行独立评分与比较分析，聚焦内容质量而非格式规范。最终目标项目：以 SQLGlot + Python + SQLite 为核心后端，前端提供在线 SQL 编译器和血缘分析画布的 SQL 血缘解析工作台。

---

## 一、逐份评分与理由

### 1. sql_lineage_workbench_technical_solution.md（最初版技术方案）

**评分：70 / 100**

**优点：**
- 技术栈选型合理，模块划分清晰，具备基本的架构意识。
- 覆盖了血缘层级、口径分析、诊断类型、开发阶段等核心维度。
- 前端页面布局和后端目录结构给出了可参考的骨架。

**不足：**
- 缺失 ScopeResolver / NameResolver 设计——字段级血缘准确性的核心链路未覆盖。
- 缺失 LineageIR 中间表示层，后端分析结果与前端图模型之间存在架构断层。
- 口径分析仅停留在维度罗列，没有给出结构化输出模型。
- 前端图谱未考虑多视图策略，复杂 SQL 下全量图不可读。
- 无 API 请求/响应契约、无诊断错误码枚举、无 Golden Case 测试样例。
- 缺少 SourceLocation（SQL/图谱联动基础）和元数据版本上下文。
- 整体更像"产品级蓝图"而非"可工程实现的方案"。

---

### 2. sql_lineage_workbench_solution_review.md（方案评审/检验报告）

**评分：82 / 100**

**优点：**
- 问题定位精准：明确指出方案缺失的 5 个关键层（作用域解析、血缘 IR、结构化口径、多视图、工程化契约）。
- 每个问题都附有具体场景说明和修正建议，不是空泛批评。
- 对"SQLGlot → React Flow 直连"的风险阐述非常到位。
- 给出了调整后的后端架构链路和模块职责重定义表，可直接指导后续版本迭代。
- 清楚区分了"必须补充"和"可暂缓"的内容，实用性强。

**不足：**
- 作为评审文档，天然无法承担方案设计职责——有些建议仍需后续版本落地。
- 部分段落存在表述重复（如 ScopeResolver 在问题1和建议目录中多次出现）。
- 缺乏对已有方案优点（如技术选型正确性）的量化肯定，整体偏批判。
- 未对不同缺陷设置优先级权重，读者需自行判断哪些最致命。

---

### 3. sql_lineage_workbench_technical_solution_v0.3.md（技术方案 v0.3）

**评分：78 / 100**

**优点：**
- 明确吸收了评审报告的 5 条核心建议，升级力度大。
- 新增了 ScopeResolver / NameResolver、LineageIR、SemanticsReport、多视图 GraphViewModel 四大关键设计。
- 模块职责表给出"做什么"和"不做什么"的明确边界，符合高内聚低耦合原则。
- LineageIR 的节点/边类型定义细致（10+ 节点类型，10+ 边类型），直接血缘与影响血缘的分离是亮点。
- Golden Case 测试集和测试目录结构已具雏形。

**不足：**
- 核心分析链路顺序为 `ScopeResolver → NameResolver → MetadataService`，与 v0.4 修正后的正确顺序 (`ScopeResolver → MetadataService → NameResolver`) 不一致——先做字段消歧再查元数据在逻辑上是错误的。
- LineageIR 仍将 semantics / diagnostics 混入同一模型，未做到彻底解耦。
- 缺少 SourceLocation 模型，SQL 与图谱联动缺乏底层支撑。
- 缺少 JSON 导入标准格式定义和元数据版本上下文。
- API 层面只有能力列表，无具体请求/响应契约。
- 诊断类型未转化为可枚举的错误码。

---

### 4. sql_lineage_workbench_technical_solution_v0.4.md（技术方案 v0.4）

**评分：85 / 100**

**优点：**
- 修正了核心链路的执行顺序（`ScopeResolver → MetadataService → NameResolver`），并附有详细的逻辑论证。
- 明确将 LineageIR 与 SemanticsReport / DiagnosticsReport 分离，由 AnalysisResult 统一聚合——架构解耦到位。
- 新增 `/api/sql/analyze` 最小请求/响应契约（含 JSON 结构），前后端协作有了稳定数据接口。
- 新增 SourceLocation 模型（行/列/偏移/文本/置信度），为 SQL 与图谱双向联动提供底层能力。
- 新增 JSON 导入标准格式（含 schema_version、metadata_version、case_sensitive 等字段），导入兼容性有保障。
- 新增 MetadataContext（版本号、大小写策略、默认库、已解析/缺失/歧义字段），血缘结果可追溯。
- 新增 DiagnosticCode 枚举（20 个错误码），诊断体系可测试、可扩展。
- P0 范围明确收缩到"简单 SQL + 基础字段血缘"，并通过示例 SQL 和验收标准表做了清晰界定。

**不足：**
- ExpressionAnalyzer 尚未独立抽取为单独模块（v0.6 才做此调整），口径分析缺乏表达式抽取前置环节。
- 多视图 GraphViewModel 仍偏概念性描述，未像其他模型一样给出 JSON 结构示例。
- 测试策略主要按类型分层，未按阶段绑定需求（v0.6 改进）。
- 部分内容与 v0.3 重复（属迭代文档的正常冗余）。
- R12/R13 顺序（口径分析 vs 表达式血缘）在 v0.6 中被证明需要调换。

---

### 5. sql_lineage_workbench_requirement_breakdown_v0.5.md（需求拆分 v0.5）

**评分：72 / 100**

**优点：**
- 将技术方案拆解为 R00-R18 共 19 个可交付需求包，每个需求包含前端/后端/数据库工作项和验收标准。
- 阶段划分 P0-P4 边界明确，P0 覆盖了从项目骨架到诊断面板的完整闭环。
- 关键依赖关系图和研发推进顺序图清晰，便于排期。
- Golden Case 集合和测试体系设计有可操作性。

**不足：**
- SourceLocation（R09）被整体放在 P1，导致 P0 中 GraphViewModel、Diagnostics、SQL Editor 缺乏位置映射模型，后续需要反向修改 P0 已完成的代码。
- 语义分析顺序：SemanticsReport（R12）先于表达式级血缘（R13），但口径分析实际依赖表达式抽取结果。
- 字段补全/select*展开/hover 被合并为一个需求（R11），三者的实现依赖和交付边界不同。
- 诊断面板（R08）出现偏晚——诊断错误码应在 API 和图谱消费之前定义。
- R04 API 契约在 P0 定义为完整响应，未考虑 MVP 可返回 partial 结果。
- 多视图图谱（R14）放在 P2，但 GraphViewModel 在 P0 未预留多视图字段，后续重构代价大。
- 测试体系（R17）横向放在末尾，而非每个需求绑定最小测试，容易导致回归困难。

---

### 6. sql_lineage_workbench_requirement_breakdown_v0.6.md（需求拆分 v0.6）

**评分：88 / 100**

**优点：**
- 在开篇处用表格清晰列出了 v0.5→v0.6 的全部调整项和原因，可追溯性极好。
- 将 SourceLocation 拆为 R09a（数据模型，P0）、R09b（精准提取，P1）、R09c（双向联动，P1），底层模型提前到位，避免反向修改。
- 将表达式分析（R13a）提前到口径分析（R12）之前，形成 `ExpressionAnalyzer → SemanticsReport` 的正确依赖链。
- 将 R11 拆为 select*展开（R11a）、completion（R11b）、hover（R11c），各自独立交付和测试。
- 将诊断拆为模型定义（R08a，P0）、诊断生成（R08b，P0）、诊断面板（R08c，P0），错误码先于消费端定义。
- 在 P0 的 GraphViewModel 中预留 view_mode / node_type / edge_type 字段，避免后续重构。
- 每个需求明确绑定最小测试，R17 从"阶段收尾"变为"全流程横向覆盖"。
- 核心数据模型（AnalysisResult、SourceLocation、GraphViewModel、DiagnosticCode、MetadataContext、JSON 导入格式）均在需求文档中以 JSON 示例给出，设计与实现对齐。
- 风险控制章节直接针对工程风险给出控制方式，不是泛泛而谈。
- P0 覆盖范围极其精确：只做单表 select + 基础字段血缘，不做复杂 CTE/子查询/窗口函数。

**不足：**
- 文档较长（1538 行），P2-P4 的需求明细相对 P0-P1 明显单薄。
- ExpressionAnalyzer 模型在正文中有路径提及但未像其他模型一样给出 JSON Schema 示例。
- 与 v0.5 的差异对比中，部分「RXX 拆为 RXXa/b/c」的说明可更突出影响面。
- P2 "多视图图谱"的具体交互规格（如节点折叠粒度、边过滤逻辑）尚缺少定义。

## 二、版本迭代趋势分析

| 维度 | 初版技术方案 | v0.3 | v0.4 | v0.5 需求拆分 | v0.6 需求拆分 | 评审报告 |
|---|---|---|---|---|---|---|
| 架构分层完整性 | 低 | 中 | 高 | — | — | — |
| 数据模型工程化 | 低 | 中 | 高 | — | 高 | — |
| API 契约稳定性 | 无 | 低 | 高 | 中 | 高 | — |
| 需求可交付性 | — | — | 中 | 中 | 高 | — |
| 依赖关系正确性 | — | — | 中 | 中 | 高 | — |
| 测试嵌入程度 | 低 | 中 | 中 | 低 | 高 | — |
| P0 范围控制 | 松 | 松 | 紧 | 中 | 紧 | — |

**整体趋势：文档质量在 5 轮迭代中持续上升。** 提升最显著的三次跳跃是：

1. **初版 → v0.3**：引入 ScopeResolver / LineageIR / SemanticsReport / 多视图，架构从"概念蓝图"升级为"分层架构"。
2. **v0.3 → v0.4**：补全 SourceLocation / MetadataContext / DiagnosticCode / API 契约 / JSON 标准格式，从"架构方案"升级为"工程方案"。
3. **v0.5 → v0.6**：精细化需求拆分、调整依赖顺序、绑定逐需求测试，从"可规划的方案"升级为"可执行的需求包"。

评审报告（82 分）作为外部质检文档，精准识别了早期版本的缺陷，其建议被 v0.3/v0.4 验证性地采纳——这构成了一个良性的"设计→评审→修正"闭环。

## 五、结论

六份文档构成了一个清晰的迭代轨迹：**初版定方向 → 评审找问题 → v0.3 补架构 → v0.4 工程化 → v0.5 拆需求 → v0.6 精化需求**。其中 v0.6 需求拆分是当前最成熟、最可直接用于开发排期的文档，v0.4 技术方案在架构层面的完整度最高。建议后续开发以 v0.6 为需求基线、v0.4 为架构参考推进。

当前方案对用户项目目标的覆盖度约为 **85%**，剩余 15% 主要集中在：表达式级血缘的模型定义尚未在需求文档中以 JSON Schema 给出、P2+ 多视图图谱的具体交互规格尚待细化、以及实际开发中 SQLGlot 对一些数仓特有语法的兼容性需在 P1 阶段实战验证。
