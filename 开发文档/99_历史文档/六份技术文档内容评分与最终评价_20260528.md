## 5. 整体可行性判断

### 5.1 技术路线可行

项目总体可行，而且技术选型方向正确。

| 模块 | 判断 |
|---|---|
| SQLGlot | 适合作为 SQL parser、AST、方言处理、基础 lineage 和格式化底座 |
| Python + FastAPI | 适合构建静态分析 API，Pydantic / OpenAPI 对契约化很友好 |
| SQLite | 适合本地/轻量元数据仓库，足以支撑表字段、注释、版本、导入历史、分析快照 |
| Monaco Editor | 适合作为在线 SQL 编辑器底座，completion / hover / diagnostics 可以自定义接入 |
| React Flow | 适合做可拖拽、可缩放、自定义节点边的血缘画布 |
| ELK.js | 适合做 DAG 自动布局，解决复杂图的初始排布 |

但必须明确：SQLGlot 不是完整产品核心。真正的产品核心应该是：

```text
SQLGlot Adapter
  → ScopeResolver
  → MetadataService
  → NameResolver
  → LineageEngine
  → ExpressionAnalyzer
  → SemanticsAnalyzer
  → DiagnosticsEngine
  → GraphBuilder
  → AnalysisResult
```

### 5.2 最大技术难点

| 难点 | 说明 | 应对 |
|---|---|---|
| 字段归属 | 无前缀字段、多表同名字段、CTE、子查询、派生表 | ScopeResolver + MetadataService + NameResolver |
| 表达式级血缘 | 聚合、case when、窗口函数、UDF、嵌套函数 | ExpressionAnalyzer 独立建模 |
| 查询口径分析 | 粒度、过滤、join、去重、指标公式不是单纯 lineage | SemanticsReport 单独建模 |
| SQL 与图谱联动 | 节点/边/诊断/口径项都要回到 SQL 片段 | SourceLocation + location_id |
| 复杂图可读性 | 全量图会失控 | GraphViewModel 多视图 + 节点折叠 + 边过滤 + 大图裁剪 |
| 元数据漂移 | 同一 SQL 在不同元数据版本下结果可能不同 | MetadataContext + metadata_version + snapshot |
| 测试回归 | 血缘结果容易“看起来对但实际错” | Golden Case + snapshot + 阶段最小测试 |

### 5.3 当前文档组合建议

推荐采用方式：

```text

需求推进主线：以 v0.6 为基准
评审原则：保留 solution_review 中的五个核心禁忌

```

## 6. 对未提到部分的合理补充

目前六份文档已经覆盖 80% 以上的产品和架构问题，但离“可以开工并少返工”还缺以下内容。

### 6.1 开发规格补充

| 缺口 | 建议 |
|---|---|
| 依赖版本 | 锁定 Python、Node、React、FastAPI、SQLGlot、Monaco、React Flow、ELK.js、SQLite 版本范围 |

| API 规格 | 补齐 `/api/sql/format`、metadata import preview/commit、table/column search、completion、hover |
| 错误响应 | 定义统一 error envelope、HTTP 状态码、partial / failed 的语义 |
| DDL | 写出 SQLite 完整 DDL、索引、唯一约束、迁移脚本 |
| 测试 fixture | 固化 sample metadata、sample SQL、expected lineage JSON |

### 6.2 数据库补充

import_changes

建议同时补充：

### 6.4 SQL 能力矩阵补充

## 7. 最终建议

最终建议是：不要再继续在“要不要做 Scope / IR / SourceLocation / Diagnostics / 多视图”这些问题上反复，答案已经明确：

```text
必须做。
```

下一步应该做的不是再写一版泛化技术方案，而是补一份“P0 开发规格说明书”，包含：

1. 精确技术栈版本；
2. 后端 Pydantic 模型；
3. 前端 TypeScript 类型；
4. SQLite DDL、索引、迁移；
5. P0 API 完整契约；
6. P0 Golden Case 输入和期望输出；
7. 前端 Workbench 布局和交互状态；
8. 最小可运行验收清单。

如果按当前文档成熟度推进，我建议采用：

```text
v0.4 作为技术架构总纲；
v0.6 作为需求实施路线；
新增 P0 开发规格说明书作为真正开发入口。
```

这样项目的落地路线会非常稳：先把简单 SQL + 元数据 + 字段血缘 + 图谱 + 诊断这条链路打穿，再逐步加入 CTE、join、select *、SourceLocation 精准联动、表达式血缘、查询口径、多视图和 diff。
